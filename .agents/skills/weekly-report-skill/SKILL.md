# 周报生成汇报 Skill

把零散周报整理为可直接汇报、可继续转 PPT 的页级 Markdown 报告，重点是：

- 内容来源于周报原文
- 输出结构严格贴近参考模板
- 缺失信息明确标注，不凭空编造

## 适用场景

- 用户要求“根据周报生成月报/工作汇报”
- 用户提供周报 Markdown，希望整理成可汇报材料
- 用户提供参考 PPT，希望按其页顺序生成汇报稿

## 输入模板（建议）

```markdown
报告类型：周报 / 月报
汇报人：<可选>
时间范围：<如 2026年4月>
周报材料：<Markdown 路径或正文>
参考模板：<可选，PPT 路径>
输出形式：汇报 Markdown
模板模式：普通模式 / 严格模板模式
```

若输入不完整，先产出首版并标注“待确认项”。

## 执行步骤

1. **校验输入**：确认已提供周报材料；若缺少时间范围或汇报人，按材料推断并标注不确定项。先显式列出 `已确认输入 / 缺失输入 / 默认假设`。
2. **识别模式**：先判断本次属于 `普通模式` 还是 `严格模板模式`。
   - `普通模式`：无参考 PPT，或用户只要结构化汇报稿。
   - `严格模板模式`：用户明确要求按 PPT 页顺序输出，且提供参考模板。
3. **解析模板**：若进入严格模板模式且提供 `.pptx`，先提取 slide 文本、页顺序、关键标题、分隔页、目录页、结束页。
4. **归并周报**：将重复周度记录按项目归并，提炼本期重点、关键进展、交付节点、问题挑战、改进措施与下期计划。
5. **建立映射**：把归并后的内容映射到模板页面；模板分隔页保留，内容页填充要点或表格；若一页素材不足，也保留该页。
6. **生成报告**：输出逐页 Markdown 汇报稿，每页一个标题，正文保持简洁、适合直接转 PPT。
7. **标注缺口**：周报没有明确支撑的信息统一放入“待确认项”，禁止臆造数据、成绩、风险结论或管理判断。

每步的最低产出要求：

| 步骤 | 最低产出 |
|---|---|
| 校验输入 | 已确认输入 / 缺失输入 / 默认假设 |
| 解析模板 | 页序列表或“退化为普通模式”的说明 |
| 归并周报 | 项目级合并结果 |
| 建立映射 | 页面与内容对应关系 |
| 生成报告 | 页级 Markdown |
| 标注缺口 | 待确认项清单 |

## 🔴 CHECKPOINT

遇到下面任一情况时，不要默默继续，先在输出中显式提醒并标出 `待确认项`：

- 用户要求“严格按模板”，但模板无法读取或页结构不清晰
- 周报材料与模板页结构明显不匹配
- 周报中缺少“问题与挑战 / 改进措施 / 下期计划”的支撑信息
- 需要推断时间范围、汇报人、项目归属，且推断存在明显不确定性

默认处理：

- 能保守输出就先给 `保守首版`
- 同时单列 `待确认项`

## 严格模板模式规则

- 只有在用户明确要求且提供参考 PPT 时启用。
- 输出必须按参考 PPT 的页顺序组织。
- 若模板页是章节分隔页（如 `PART 01`、`PART 02`），也必须保留。
- 若模板页是封面、目录、结束页，应生成对应页级内容。
- 若模板中的某页在周报里找不到足够素材，保留该页并标注“待确认项”。
- 若模板可读但标题含糊，优先保留页序和页面角色，不擅自重命名整份结构。
- 若模板无法可靠提取文本，退化为普通模式，并明确说明退化原因。

## 周报归并规则

- 相同项目在不同周出现时，先合并再总结，不按原文简单堆叠。
- 将“上线 / 提测 / 联调 / 开发完成 / 评审 / 对接”视为关键进展信号。
- 将“计划”章节单独映射到“下期计划”相关页面。
- 如无明确“问题与挑战/改进措施”，只允许基于现有内容做谨慎抽象，并注明依据不足。
- 若同一事项在多周重复出现，优先提炼“阶段推进”而不是重复列条目。
- 若项目归属不清，先按周报原词保留，不强行重组到错误项目名下。

## 标准输出结构

```markdown
# 第1页｜封面
- 标题：<时间范围><报告类型>
- 汇报人：<姓名>
- 日期：<输出日期>

# 第2页｜目录
- 01 工作总结
- 02 问题与挑战
- 03 改进措施
- 04 下月计划

# 第3页｜PART 01 工作总结

# 第4页｜工作总结
| 项目 | 主要事项 | 进度 | 总结 |
|---|---|---|---|

# 第5页｜工作总结
| 项目 | 主要事项 | 进度 | 总结 |
|---|---|---|---|

# 第6页｜问题与挑战
- 问题 1：...
- 问题 2：...
- 待确认项：...

# 第7页｜改进措施
- 改进 1：...
- 改进 2：...

# 第8页｜下月计划
| 项目 | 计划事项 | 时间/节点 |
|---|---|---|

# 第9页｜结束页
- 汇报人：<姓名>
- 致谢语：谢谢欣赏
```

当参考模板页数更多时，继续按模板页顺序扩展，不擅自省略模板页。

## 写作规则

- 先总结，再展开，不复读原始周报。
- 保留项目名称与关键业务名词，避免改写失真。
- 句子要简洁，适合投屏展示与口头汇报。
- 表格字段名优先复用参考模板的命名方式。
- 没有依据的量化指标不要编造。
- 没有依据时，不把“忙”“推进中”包装成成果亮点。
- 不把普通开发动作夸大为业务成绩或管理结论。

## 失败分支与兜底

使用三段式表格，不要只写正向流程。

| 触发条件 | 一线处理 | 仍失败时输出 |
|---|---|---|
| 周报材料缺失 | 先停止生成正文，整理用户已给上下文 | 明确列出 `待补充信息`，不生成伪完整汇报 |
| 时间范围不明确 | 优先从周报标题、日期、文件名推断 | 标注“时间范围待确认”，继续输出保守版 |
| 严格模板模式但 PPT 无法读取 | 先尝试按用户描述或文件名判断模板角色 | 退化为普通模式，并说明未能按模板精确还原 |
| 模板页数过多而周报素材不足 | 保留页序，只填已有内容 | 对空缺页标注“待确认项”，不擅自删页 |
| 问题与挑战缺乏依据 | 只做谨慎抽象，限定为“可能存在的挑战” | 明确写“依据不足，待补充” |
| 下期计划信息缺失 | 只抽取周报中明确的计划项 | 若没有明确计划，保留页面并标注待确认 |

## 生成后自检

生成完成后，按下面清单逐条自检；若不满足，先修再交付：

- [ ] 是否先区分了普通模式 / 严格模板模式
- [ ] 是否给出了已确认输入 / 缺失输入 / 默认假设
- [ ] 是否按模板页顺序或标准结构输出
- [ ] 是否完成了跨周项目归并，而不是逐条搬运
- [ ] 是否把计划内容单独放入“下期计划”
- [ ] 是否明确标注了待确认项
- [ ] 是否没有编造量化指标、成果或管理判断
- [ ] 是否没有把原文流水账直接原样展开

## 反例黑名单

- 不要把周报原文逐条复制后改几个词就当汇报稿。
- 不要为了“看起来完整”编造数据、结果、风险或改进措施。
- 不要在严格模板模式下擅自省略模板页。
- 不要把“计划做”写成“已经完成”。
- 不要把多个项目的事项混在同一页里导致归属不清。
- 不要输出空泛标题，如“其他事项”“若干进展”，除非模板本来如此。

## 质量检查清单

- [ ] 输出是否按模板页顺序组织
- [ ] 周报重复项目是否已归并
- [ ] `计划`内容是否单独进入“下期计划”页
- [ ] 缺失信息是否明确标注“待确认项”
- [ ] 是否避免了原文逐条照抄
