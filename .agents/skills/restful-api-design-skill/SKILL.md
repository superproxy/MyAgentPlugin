# RESTful API 设计与校验 Skill

先定义统一 API 规范，再据此做设计、自检、对照检查与生成，重点是：

- 资源建模清晰
- 文档与 OpenAPI 一致
- 检查结果可直接落地修正
- 生成结果可作为研发起点

## 适用场景

- 用户要求“设计一个 RESTful API”
- 用户要求“先出 API 规范 / OpenAPI 草案”
- 用户要求“检查 API 设计是否符合 REST 规范”
- 用户要求“检查代码/文档与 API 规范是否一致”
- 用户要求“生成接口示例、测试草案、服务端骨架”

## 模式说明

### define
从业务需求出发，先定义 RESTful API 规范，再输出可评审的接口设计结果。

### check
对现有规范做自检，并对用户明确提供的代码、文档做定向一致性检查，输出问题与修正建议。

### generate
基于规范化设计，生成完整草案包：设计文档、OpenAPI、示例、测试草案、服务端骨架。

默认不要做仓库级泛扫描；只有在用户明确提供规范、代码、文档路径时，才做定向比对。

## 输入模板（建议）

```markdown
模式：define / check / generate
API 名称：<名称>
业务目标：<为什么需要这个 API>
资源模型：<核心资源、子资源、资源关系>

鉴权方式：<如 Bearer Token / Session / API Key>
分页/过滤/排序要求：<可选>
错误处理约定：<可选>
版本策略：<可选>
```

若输入不完整，先产出首版并标注“待确认项”；不要为了补全格式而臆造业务约束。

## RESTful API 规范定义

这部分是本 skill 的统一内置标准。define / check / generate 三种模式都必须先以此为准。

### 1. 资源建模

- 优先抽象资源，不优先抽象动作。
- 路径名默认使用名词，集合资源默认使用复数，如 `/users`、`/orders`。
- 只有存在明确从属关系时才使用子资源，如 `/orders/{orderId}/items`。
- 不把筛选条件、展示态字段、前端视图概念直接建成资源。

### 2. URI 规则

- 路径表达资源层级，不表达操作流程。
- 避免使用动词型路径，如 `/getUsers`、`/createOrder`。
- 同一领域下保持命名风格一致：大小写、分隔符、复数形式统一。
- 批量操作、状态变更、导出等偏动作语义场景，要明确说明为何不直接使用基础资源操作。

### 3. HTTP Method 语义

- `GET`：查询资源，不产生副作用。
- `POST`：创建资源或触发非幂等动作。
- `PUT`：整体替换资源，默认幂等。
- `PATCH`：局部更新资源。
- `DELETE`：删除资源，默认幂等。
- 不混用方法语义，例如不要用 `POST` 代替普通查询。

### 4. 状态码规则

- `200`：成功返回。
- `201`：创建成功。
- `202`：已受理但未完成。
- `204`：成功但无响应体。
- `400`：请求格式或参数错误。
- `401`：未认证。
- `403`：已认证但无权限。
- `404`：资源不存在。
- `409`：资源冲突、重复提交、版本冲突。
- `422`：语义校验失败。
- `500`：服务端未预期错误。
- 如使用其他状态码，必须说明适用条件。

### 5. 错误响应模型

错误响应默认统一结构，至少包含：
- `code`
- `message`
- `details`
- `requestId`

要求：
- `code` 可稳定引用，适合前后端联调和告警归类。
- `message` 面向开发者，语义清晰。
- `details` 用于字段级错误、约束冲突或上下文补充。
- `requestId` 用于链路追踪。

### 6. 请求与响应 Schema

- 明确每个字段的类型、是否必填、枚举、格式、默认值、示例。
- 时间字段统一格式，如 RFC 3339。
- ID 字段命名保持一致，如 `userId`、`orderId`。
- 列表响应要清楚区分数据列表与分页元信息。
- 对外暴露字段只保留消费方需要的内容，敏感字段默认不透出。

### 7. 分页 / 过滤 / 排序

- 若列表数据可能增长，应定义分页策略。
- 分页参数命名保持统一，如 `page` / `pageSize` 或 `cursor` / `limit`。
- 过滤字段应与资源属性一致，避免模糊缩写。
- 排序字段与排序方向要显式定义，如 `sortBy`、`sortOrder`。
- 响应中明确返回分页元信息或游标信息。

### 8. 幂等性

- `PUT`、`DELETE` 默认按幂等操作设计。
- 对可能重复提交的写操作，必要时定义幂等键策略。
- 任何非幂等写操作都要在文档中明确风险与行为预期。

### 9. 安全与鉴权

- 明确认证方式、授权边界、资源访问控制规则。
- 涉及敏感信息时，说明脱敏、最小暴露、审计要求。
- 对管理接口与普通用户接口区分权限模型。
- 必要时说明限流、防重放、签名等约束。

### 10. 版本与兼容性

- 明确是否使用版本化路径、Header 版本或兼容演进策略。
- 新增字段应默认向后兼容；废弃字段要标记迁移方式。
- 破坏性变更必须单独提示影响范围。

### 11. 文档一致性

- Markdown 设计文档与 OpenAPI 3.1 草案必须一一对应。
- endpoint、参数、schema、状态码、错误码描述不能互相冲突。
- 示例请求/响应必须与 schema 一致。

## 执行步骤

### define 模式

1. **识别边界**：提炼业务目标、角色、资源对象、范围边界。
2. **定义规范**：先给出本 API 适用的设计规范摘要。
3. **建立资源模型**：梳理主资源、子资源、资源关系。
4. **映射接口**：将核心操作映射为 method + path。
5. **补齐契约**：定义参数、请求体、响应体、状态码、错误模型。
6. **补齐治理项**：定义鉴权、分页、过滤、排序、版本与兼容策略。
7. **双输出**：产出 Markdown API 设计文档与 OpenAPI 3.1 草案。
8. **列出待确认项**：把不确定点单独列出。

### check 模式

1. **读取材料**：只检查用户明确提供的规范、代码、文档。
2. **规范自检**：先按“RESTful API 规范定义”做设计自检。
3. **代码对照**：若提供代码，检查路由、method、参数、状态码、错误结构是否一致。
4. **文档对照**：若提供文档，检查命名、字段、示例、约束是否一致。
5. **归类问题**：按规范问题、代码不一致、文档不一致、风险与歧义输出。
6. **给出修正**：每个问题都要给出证据、影响、修正建议。
7. **必要时补规范**：若用户需要，给出建议修正版规范或修正版 endpoint 草案。

### generate 模式

1. **规范化设计**：先执行 define 模式的规范化过程。
2. **输出设计文档**：生成 Markdown API 设计文档。
3. **输出 OpenAPI**：生成 OpenAPI 3.1 草案。
4. **生成示例**：输出请求/响应示例。
5. **生成测试草案**：覆盖成功、失败、边界、鉴权、幂等、冲突等场景。
6. **生成服务端骨架**：按用户指定技术栈给出 route/controller、service、DTO/schema、error handling、auth middleware、test outline。
7. **标注假设**：把假设、待补充项、实现前依赖单独列出。

### OpenAPI 3.1 草案模板

```yaml
openapi: 3.1.0
info:
  title: <API Name>
  version: 1.0.0
  description: <API Description>
servers:
  - url: https://api.example.com
tags:
  - name: <Resource>
paths: {}
components:
  schemas: {}
  parameters: {}
  responses: {}
  securitySchemes: {}
security: []
```

要求：
- `paths` 中的接口必须与 Markdown 设计文档一致。
- `components.schemas` 要覆盖请求体、响应体、错误模型。
- `securitySchemes` 与 `security` 必须与鉴权说明一致。

## 输出规则

- 先给规范摘要，再给具体 endpoint 设计。
- 若用户未提供代码或文档，禁止自行扩展为仓库级扫描任务。
- 若输入缺少关键业务信息，正常产出首版，但必须在“待确认项”中标出。
- 不要把实现细节伪装成业务规范；设计与骨架要分层输出。
- check 模式不能只列问题，必须附修正建议。
- generate 模式不能只给 OpenAPI，必须同时给示例、测试草案、服务端骨架。
