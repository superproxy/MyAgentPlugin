# 基于用例的产品需求 Skill

把零散需求整理为可评审、可开发、可验收的产品需求文档，核心是：

- 以用户角色为中心
- 以用户操控动作为主线
- 以用例规约保障可落地

## 适用场景

- 用户要求“根据需求生成产品需求文档”
- 用户强调“按用例维度描述功能”
- 需求来源是企划书、会议纪要、口述流程、招投标文档

## 输入模板（建议）

```markdown
项目/产品名称：<名称>
业务背景：<为什么做>
目标用户：<角色列表>
范围说明：<本期包含/不包含>
需求材料：<原始文本、链接或附件摘要>
约束条件：<平台、兼容、性能、安全、合规>
```

若输入不完整，先产出首版并标注“待确认项”。

## 执行步骤

1. **提炼边界**：明确目标、角色、设备/平台范围、版本范围。
2. **识别角色与场景**：把“谁在什么场景下要完成什么目标”整理成用例候选。
3. **用例化功能**：每个功能都改写为“用户操控动作 -> 系统响应结果”。
4. **补全规约**：为每个关键用例补齐前置条件、触发条件、主成功场景、异常分支、后置条件。
5. **定义验收**：把模糊描述转成可测试的验收标准（功能、兼容、体验）。
6. **标注优先级**：按 P0/P1/P2 排序，便于排期。

## 用例写作规则（重点）

- 用例名称使用“动词 + 业务对象”，如“连接蓝牙设备”“切换音源”。
- 主成功场景只写用户可感知动作与系统可观察反馈，避免实现细节。
- 每条步骤保持单一动作，不把多个动作塞进一条。
- 异常分支必须可触发、可验证，如“设备离线”“权限被拒绝”。
- 验收标准尽量使用“Given/When/Then”表达。

## 标准输出结构

```markdown
## 1. 文档概述
### 1.1 背景与目标
### 1.2 术语与定义
### 1.3 范围（In Scope / Out of Scope）

## 2. 用户与场景
### 2.1 角色画像
### 2.2 核心使用场景

## 3. 用例总览
| 用例ID | 用例名称 | 角色 | 价值目标 | 优先级 |
|---|---|---|---|---|

## 4. 详细用例规约
### UC-01 <用例名称>
- 目标：
- 参与者：
- 前置条件：
- 触发条件：
- 主成功场景（用户操控 -> 系统响应）：
  1. 用户...
  2. 系统...
- 异常/扩展流程：
  - E1 ...
  - E2 ...
- 后置条件：
- 业务规则：
- 验收标准（Given/When/Then）：

## 5. 功能需求清单（按模块）
- FR-001 ...
- FR-002 ...

## 6. 非功能需求
- 兼容性：
- 性能：
- 可用性与可访问性：
- 安全与隐私：

## 7. 版本规划与依赖
- 本期范围：
- 后续规划：
- 外部依赖与风险：

## 8. 待确认项
- Q1 ...
- Q2 ...
```

## 质量检查清单

- [ ] 每个 P0 功能都有对应详细用例
- [ ] 每个用例都有“用户操控 -> 系统响应”步骤
- [ ] 每个关键用例都有异常分支
- [ ] 验收标准可直接用于测试用例设计
- [ ] 范围边界与不做项写清楚
